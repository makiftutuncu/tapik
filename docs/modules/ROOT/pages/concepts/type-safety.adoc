= Type Safety in tapik

tapik's primary goal is to keep HTTP contracts type-safe from declaration to runtime. The DSL models every piece of an endpoint-parameters, headers, bodies, and response branches-with immutable, strongly-typed structures. Generators rely on these structures to emit Kotlin that cannot drift from the original definition.

== Building Blocks

|===
| Concept | Interface | Purpose | Common Aliases

| Heterogeneous tuple (product)
| +AllOf+
| Represents *all* items that must exist together
| +Parameters+, +Headers+, +Outputs+, +HeaderValues+, +Responses+

| Discriminated union (sum)
| +OneOf+
| Represents *one* branch chosen at runtime
| +OneOf2+, +OneOf3+, ... up to +OneOf10+
|===

=== AllOf and Tuple Aliases

+AllOf+ is a family of interfaces (+AllOf0+ ... +AllOf10+) with +itemN+ accessors. Concrete tapik types reuse these through the +Tuple+ facade so you do not manipulate +AllOf+ directly.

[source,kotlin]
----
typealias Headers3<H1, H2, H3> =
    Tuple3<Header<*>, Header<H1>, Header<H2>, Header<H3>>
----

Because +Headers3+ implements +AllOf4+, the compiler knows:

* +headers.item1+ is the first +Header<*>+.
* +headers.item2+ is +Header<H1>+, preserving the codec type.
* Iteration is still possible via the +Listable+ interface.

tapik ships aliases for each arity so you can declare headers, parameters, and responses without manual tuple plumbing:

|===
| Alias | Description

| +Parameters0+ ... +Parameters10+
| Path/query parameter definitions

| +Headers0+ ... +Headers10+
| Request header definitions

| +HeaderValues0+ ... +HeaderValues10+
| Decoded inbound header values

| +Outputs0+ ... +Outputs10+
| Ordered list of +(StatusMatcher, Headers, Body)+ combinations

| +Response0+ / +ResponseWithoutBody0+ families
| Runtime response wrappers used internally by adapters and interpreters
|===

=== OneOf and Conditional Responses

+OneOf+ models exclusive branches. Each arity exposes inline +OptionN+ value classes and a +select+ helper for exhaustive handling.

[source,kotlin]
----
fun interpret(result: OneOf3<Response0<User>, ResponseWithoutBody0, ResponseWithoutBody0>) =
    result.select(
        when1 = { response -> "Found user ${response.body.id}" },
        when2 = { "User missing" },
        when3 = { "User deleted" }
    )
----

Generators still rely on these unions internally, but generated public APIs no longer expose them directly. Instead, code generation derives endpoint-specific response contracts:

* *Single-output endpoints* generate one +data class+ or +data object+.
* *Multi-output endpoints* generate a sealed +Response+ hierarchy with one subtype per output branch.
* *Adapters and interpreters* continue to use +OneOf+ and +ResponseX+ under the hood when converting between HTTP and generated contracts.

=== AllOf + OneOf Together

+Outputs+ is an +AllOf+ tuple of +Output+ definitions. Code generation turns that metadata into endpoint-specific response contracts, while the runtime still uses +ResponseX+ and +OneOf+ internally for decoding and adaptation. This combination gives tapik strong guarantees:

. The number and order of outputs in the DSL matches the generated code and Markdown documentation.
. Header codecs are preserved all the way to runtime decoding (+decodeHeadersN+).
. Adding a new output changes the generated sealed hierarchy, which becomes a compile-time change for every consumer.

[mermaid]
----
flowchart TD
    DSL["HttpEndpoint DSL"] --> Tuple["Tuple / AllOf metadata"]
    Tuple --> Outputs["Outputs tuple"]
    Outputs --> Contract["Generated endpoint contract"]
    Contract -->|single| PublicSingle["data class / data object"]
    Contract -->|multiple| PublicUnion["sealed Response hierarchy"]
    PublicSingle --> Client["Generated client"]
    PublicUnion --> Client
    PublicSingle --> Server["Generated controller"]
    PublicUnion --> Server
    Client --> Runtime["tapik ResponseX / OneOf internals"]
    Server --> Runtime
----

== Tips for Working with These Types

. *Prefer aliases*: use +Headers3+ instead of wiring +Tuple3+ manually. The aliases pick correct variance and marker interfaces.
. *Let the DSL infer tuples*: +input(header.uuid("X-Request-Id"))+ returns +Input<Headers1<UUID>, EmptyBody>+ automatically-no explicit types required.
. *Use generated response hierarchies*: in clients and servers, branch on the generated endpoint-specific +Response+ subtypes rather than runtime +OneOf+ wrappers.
. *Trust constructor validation*: generated responses validate non-exact status matchers at construction time, so invalid combinations fail fast.
. *Leverage codecs*: bodies and headers carry codecs, so even custom +AllOf+ combinations continue to encode/decode data safely.

Understanding +AllOf+ and +OneOf+ is still key to appreciating tapik: the DSL describes APIs once, and the type system guarantees that every generated artifact agrees with the definition. Those runtime types now sit behind endpoint-specific generated contracts instead of leaking into consumer code. Take a look at the runtime helpers (+EncodeHeadersMethods.kt+, +DecodeHeadersMethods.kt+, +Responses.kt+) if you want to explore further.
