= Spring WebMVC Controllers

The +spring-webmvc+ generator turns tapik endpoints into Spring MVC controller contracts. Generated server signatures now return endpoint-specific response types instead of tapik's internal +ResponseX+ and +OneOf+ wrappers.

== Generated Structure

* *Location*: +build/generated/sources/tapik/main/kotlin/<package>/<SourceFile>Server.kt+
* *Aggregate interface name*: +<SourceFile>Server+
* *Endpoint contract name*: +<SourceFile>Endpoints.<EndpointProperty>+
* *Annotations*: Each method is decorated with the appropriate +@GetMapping+, +@PostMapping+, ... annotations derived from the endpoint metadata, including +path+, +consumes+, and +produces+ attributes.
* *Parameters*: Path variables receive +@PathVariable+, query parameters use +@RequestParam+, headers use +@RequestHeader+, and bodies use +@RequestBody+. Default values defined in the DSL are carried into the generated signatures.
* *Return type*: Methods return the generated endpoint response type. Single-output endpoints get one +data class+ or +data object+; multi-output endpoints get a sealed +Response+ hierarchy.

Example excerpt:

[source,kotlin]
----
interface ProductEndpointsServer : ProductEndpoints.GetProduct.Server

interface ProductEndpoints {
    interface GetProduct {
        data class Ok(val response: ProductView)

        interface Server : dev.akif.tapik.Helpers {
            @GetMapping(
                path = ["/products/{productId}"],
                produces = ["application/json"]
            )
            fun getProduct(
                @PathVariable(name = "productId") productId: UUID,
                @RequestParam(name = "locale", required = false) locale: String = "en-US",
                @RequestHeader(name = "X-Request-Id") xRequestId: UUID
            ): Ok
        }
    }
}
----

== Implementing Controllers

Wire an implementation in your Spring application:

[source,kotlin]
----
@Component
class ProductHandler : ProductEndpointsServer {
    override fun getProduct(
        productId: UUID,
        locale: String,
        xRequestId: UUID
    ): ProductEndpoints.GetProduct.Ok {
        val product = service.lookup(productId, locale)
        return ProductEndpoints.GetProduct.Ok(response = product)
    }
}
----

For multiple outputs, return the matching sealed subtype. This keeps the implementation exhaustive without exposing transport wrappers in your application code.

== Rendering Spring Responses

The generated server contract stops at the typed response model. Rendering those responses as Spring +ResponseEntity+ values is intentionally left to a separate adapter layer.

[source,kotlin]
----
@RestController
class ProductController(
    private val handler: ProductHandler
) {
    @GetMapping("/products/{productId}")
    fun getProduct(
        productId: UUID,
        locale: String,
        xRequestId: UUID
    ): ResponseEntity<Any?> {
        val response = handler.getProduct(productId, locale, xRequestId)
        // translate ProductEndpoints.GetProduct.Ok into ResponseEntity here
    }
}
----

This keeps the generated server API focused on the contract itself. If you want automatic Spring rendering later, build that as a dedicated adapter on top of the generated response hierarchy.

== Status and Header Validation

Non-exact status matchers still validate the runtime status value in the generated response constructors.

== Tips for Server Implementations

. *Reuse DSL metadata*: The generated interfaces still reference the +HttpEndpoint+ instances, so you can lean on +.outputs+ or +.parameters+ inside your implementation for additional logic.
. *Combine with Spring validation*: Add validation annotations such as +@Valid+ or +@NotBlank+ in wrapper controllers when you need extra constraints.
. *Return endpoint-specific responses*: Construct the generated +data class+ or +data object+ directly instead of assembling tapik runtime wrappers by hand.
. *Unit-test with generated contracts*: Test the generated response hierarchy directly and add adapter-level tests in your Spring translation layer.

== Customising Routing

The generator chooses method-level annotations based on the HTTP verb. For uncommon verbs such as +OPTIONS+ or +TRACE+, it emits +@RequestMapping(method = [RequestMethod.VERB], path = [...])+. If you need additional attributes such as security annotations or +@CrossOrigin+ settings, declare them in your concrete controller implementation or wrapper controller.
