= Spring RestClient Clients

The +spring-restclient+ generator produces Kotlin interfaces that call HTTP endpoints through the Spring Framework +RestClient+ API. Generated sources now expose endpoint-specific contracts instead of tapik's internal +ResponseX+ and +OneOf+ runtime wrappers.

== Generated Structure

* *Location*: +build/generated/sources/tapik/main/kotlin/<package>/<SourceFile>Client.kt+
* *Aggregate interface name*: +<SourceFile>Client+
* *Endpoint contract name*: +<SourceFile>Endpoints.<EndpointProperty>+
* *Base interface*: +RestClientBasedClient+, which exposes:
** +val restClient: RestClient+
** +val interpreter: RestClientInterpreter+ (lazy wrapper around +restClient+)
* *Method signature*: one function per endpoint property, using descriptive parameter names derived from the DSL and returning the endpoint contract's generated response type.

Example excerpt:

[source,kotlin]
----
interface ProductEndpointsClient : ProductEndpoints.GetProduct.Client

interface ProductEndpoints {
    interface GetProduct {
    data class Ok(val response: ProductView) 

        interface Client : RestClientBasedClient {
            fun getProduct(
                productId: UUID,
                locale: String = "en-US",
                xRequestId: UUID
            ): Ok {
                // generated HTTP call and decoding
                return Ok(response = decodedBody)
            }
        }
    }
}
----

[NOTE]
====
Names and types are inferred from your endpoint declarations: path/query parameters become function parameters, headers with codecs translate into arguments, and bodies map to encoded values.
====

== Response Types

|===
| Endpoint outputs | Generated return type

| Single output
| A direct endpoint-specific +data class+ or +data object+

| Multiple outputs
| A nested sealed +Response+ hierarchy inside the endpoint contract
|===

For an endpoint with multiple outputs, the generated contract looks like this:

[source,kotlin]
----
interface UsersEndpoints {
    interface Create {
        sealed class Response(open val status: Status) {
            data class Created(
                val response: UserResponse,
                val location: URI
            ) : Response(Status.Created)

            data class BadRequest(
                val error: ApiError
            ) : Response(Status.BadRequest)
        }
    }
}
----

This keeps client code exhaustive without exposing internal transport wrappers. Consumers branch on generated response variants directly.

== Header and Body Handling

* Headers are encoded via the +Input.encodeInputHeaders+ helpers from +core+; generated clients decode them and populate the matching response constructor fields.
* Bodies use the codec attached to the +Body+ definition (for example, Jackson-based JSON codecs). Any +CodecFailure+ surfaces as an +IllegalStateException+ in generated code so mismatches fail fast.
* +RestClientInterpreter+ validates response status codes and media types before the generator attempts to decode bodies, surfacing detailed +RestClientResponseException+ messages when mismatches occur.

== Instantiating Clients

Create a Spring-managed implementation by delegating to the generated interface:

[source,kotlin]
----
@Component
class HttpProductClient(
    override val restClient: RestClient
) : ProductEndpointsClient
----

+RestClientBasedClient+ provides +interpreter+ automatically, so implementations only supply the +RestClient+. You may inject a configured +RestClient.Builder+ or reuse the application-wide client.

== Testing Tips

. Drive endpoint behaviour through unit tests on the DSL before running code generation.
. Use Spring's +MockRestServiceServer+ to verify generated clients: intercept outgoing requests, assert the expected path/headers/body, and feed back fixtures that satisfy your endpoint outputs.
. For multi-output endpoints, assert on the generated sealed response variants rather than internal +OneOf+ wrappers.
. If you customise codecs, add integration tests to ensure +RestClientInterpreter+ can successfully decode the wire format.

== Customisation Hooks

* Provide your own +RestClient+ (set timeouts, interceptors, authentication) while reusing generated code.
* Override functions in the generated interface by writing extension functions or wrapping the generated interface inside a higher-level service.
* Add new generators for other HTTP client stacks by following the steps in xref:code-generation/index.adoc#extending-tapik[Code Generation Overview]; the Spring client serves as a realistic template.
